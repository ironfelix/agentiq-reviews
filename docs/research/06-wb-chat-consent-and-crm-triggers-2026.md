# WB Chat: Consent, Personal Data, and CRM Triggers (2026)

_Last checked: 2026-02-12_

## TL;DR for agentiq

Для `WB чата` нужно разделять два разных согласия:

1. **Согласие на рекламу** (38-ФЗ ст. 18):
- Для рекламных сообщений по каналам связи требуется предварительное согласие адресата.
- Это требование действует независимо от вопроса, есть ли у вас ФИО/телефон в явном виде.

2. **Согласие на обработку ПД для маркетинга** (152-ФЗ ст. 15):
- Для прямых контактов в целях продвижения товаров через средства связи нужно предварительное согласие субъекта ПД.

**Вывод:**
- Сервисные ответы в рамках обращения в чате WB можно строить без отдельного marketing-consent.
- Триггеры в стиле «стимулировать повторную покупку» в чате WB как маркетинговые сообщения без отдельного согласия и вне контекста обращения несут высокий регуляторный и платформенный риск.

## Что важно именно для вопроса «в чате нет персональных данных»

Это допущение для WB API на практике неверно:
- В событиях чата есть `clientID` и `clientName`.
- В сообщениях могут быть вложения/текст с данными покупателя.
- В оферте WB прямо закреплено: нельзя собирать, хранить и распространять персональные данные покупателей в чате.

С точки зрения 152-ФЗ персональные данные — это любая информация о прямо или косвенно определяемом физлице (ст. 3). Идентификатор + контекст покупки обычно попадает в эту рамку.

## Позиция WB по чату (платформенные ограничения)

По действующим правилам WB:
- Чат инициирует покупатель (API: buyer always starts the chat).
- В чате нужно отвечать на обращения аргументированно и в срок.
- В условиях 9.3.5 есть требования к содержанию общения, включая запрет:
  - мотивировать покупателя перейти на другую площадку/мессенджер (с исключением передачи своих контактов),
  - собирать/хранить/распространять ПД покупателей,
  - рекламировать или упоминать сторонние ресурсы/организации/продавцов, включая внешние ссылки.
- За нарушения возможна блокировка отправки новых сообщений в чате до удаления нарушающих ответов; при повторных грубых нарушениях есть риск блокирующих мер к кабинету.

## Можно ли делать триггеры после обращения

Да, но нужно разделять типы триггеров:

## Типы сценариев и приоритеты (P0-P3)

Ниже практическая приоритизация для product roadmap и для policy-layer (в первую очередь применимо к `WB чату`):

- **P0 (обязательно, core сервиса)**: Service, Recovery
- **P1 (сильно влияет на удержание, обычно безопасно как сервис)**: Trust & CX, Education
- **P2 (дает деньги, но требует строгих правил и модерации)**: Service-Mixed
- **P3 (маркетинг; в WB-чате высокий комплаенс/платформенный риск)**: Reactivation, Promo, Loyalty/Retention

Краткие определения:
- **Service**: статус/уточнение/закрытие кейса, возврат/обмен.
- **Recovery**: антикризис после негатива (плохой отзыв, претензия, сорванный срок, дефект).
- **Service-Mixed**: сервис + релевантная рекомендация в рамках решения кейса (например, альтернативный артикул).
- **Education**: FAQ/инструкции/уход/совместимость/подбор.
- **Trust & CX**: оценка решения (CSAT/NPS), просьба обновить отзыв без вознаграждения.
- **Reactivation**: “разбудить” покупателя без новых обращений.
- **Promo**: акции/промокоды/price-drop/back-in-stock/сезонные офферы.
- **Loyalty/Retention**: повторные покупки, подборки, персональные предложения.

### A. Допустимые сервисные триггеры в WB-чате (без marketing-consent)

- Подтверждение получения обращения.
- Запрос недостающих данных для решения кейса.
- Статус по претензии/возврату.
- Сообщение о закрытии кейса.
- Follow-up по качеству решения по конкретному кейсу (без рекламы).

Условие: сообщение функционально связано с исходным обращением и не является рекламой.

### B. Рискованные/нежелательные в WB-чате

- Кросс-сейл/апсейл «купи еще» после решения кейса.
- Промокоды «на следующую покупку» в чате как массовая тактика.
- Реактивационные цепочки без нового клиентского запроса.

Почему риск:
- Вероятность квалификации как реклама по каналу связи.
- Требование предварительного согласия по рекламе/ПД-маркетингу.
- Чат WB по правилам заточен под сервисное взаимодействие, а не CRM-рассылки.

### C. Где делать CRM-маркетинг правильно

- Во внешних каналах, где есть зафиксированное согласие на рекламу и на обработку ПД в маркетинговых целях.
- С учетом обязательного stop-механизма и доказуемости согласий.

## Когда P3 допустим (практическая логика)

**Базовое правило:** `P3` в `WB чате` по умолчанию запрещен. Разрешение возможно только в ограниченных “окнах”, когда сообщение не выглядит как инициативная рассылка и связано с контекстом обращения.

### Окно 1: Ответ на прямой запрос покупателя (pull-marketing)

Разрешать `P3` только если выполняется все:
- Последнее сообщение в чате от покупателя и содержит явный intent (например: “есть скидка/промокод?”, “посоветуйте аналог”, “что еще подойдет?”, “будут акции?”).
- Сообщение-ответ одноразовое (без цепочки и без повторной реактивации по этому же чату).
- Нет внешних ссылок, упоминаний сторонних ресурсов/продавцов.
- Нет “давления” (например: “успей”, “только сегодня”, “последний шанс”).

### Окно 2: Компенсация/урегулирование в рамках проблемного кейса (settlement)

Разрешать `P3` только если:
- Есть активный кейс типа дефект/ошибка/срыв срока/претензия, и промокод/скидка используется как инструмент урегулирования.
- Нет связки с отзывом (нельзя: “за отзыв/за 5 звезд дадим...”).
- Фиксируется reason-code: `compensation` + привязка к конкретному кейсу.

### Всё остальное (включая массовые “реактивации”) выносить во внешний контур

Реактивации/промо/лояльность делаются во внешних каналах при наличии согласий на рекламу/маркетинг.

## Практическая политика для agentiq (рекомендуемая)

Ввести в продукте жесткий policy-layer:

- `channel=wb_chat` + `message_type=service` -> allow.
- `channel=wb_chat` + `message_type=marketing` -> deny (по умолчанию).
- `channel=wb_chat` + `message_type=marketing` -> allow только если `marketing_window in {customer_request, compensation_case}`.
- `channel=wb_chat` + `message_type=marketing` -> allow только если `frequency_cap_ok=true` (например, не более 1 P3 на чат за 30-90 дней).
- `channel=wb_chat` + `message_type=marketing` -> allow только если `chat_state=active` (есть сообщение покупателя за последние N часов/дней) и `last_sender=buyer` для окна `customer_request`.
- Любой текст с промо-лексикой (`скидка`, `акция`, `купи еще`, `промокод`, `подпишись`) в WB-чате -> ручная модерация/блок.
- Любые ссылки на внешние ресурсы в WB-чате -> блок, кроме разрешенных контактных данных по правилам WB.
- Хранить audit trail: источник триггера, шаблон, оператор, время, версия правил.

## Ответ на ваш исходный тезис

Тезис: «в WB-чате нет персональных данных, поэтому можно делать триггеры на повторные покупки».  
Корректировка:
- В API чата есть поля, которые относятся к данным покупателя (`clientID`, `clientName`) и контекст заказа.
- Даже если бы ПД не было, для рекламных сообщений через связь всё равно остается требование предварительного согласия по 38-ФЗ ст. 18.
- Плюс действуют прямые ограничения оферты WB по использованию чата.

Итог: в чате WB безопасно и масштабируемо автоматизировать **service lifecycle**, а не **outbound CRM-повторные продажи**.
